雙瓶頸統一公式

TPS = min(TPS_mem, TPS_comp)

LLM 推論的速度上限由兩條路徑中較慢的一條決定：記憶體路徑（BW 有多快）或計算路徑（Core 有多強）。v4 在 v3 基礎上引入 reuseFactor 和 n_memory，使記憶體路徑能正確表達不同裝置的差異。

Memory Path（v4）

BPT = weight + KV + activation
EBW = BW × n_memory
TPS_mem = EBW / (BPT / reuse)

weight = P×Q×weightPassFactor；KV 隨輸入長度 L 增長；activation = 0.05×weight。reuseFactor 越高，每次有效搬運量越少，TPS_mem 越高。這是 NPU/iGPU 速度快於 CPU 的根本原因。

Compute Path 與 Core Adequacy

C_req = BW / tpc
TPS_comp (CPU) = C × tpc / (P×Q) × η
SIMDtpcCPU 需 cores
AVX212 GB/sBW÷12
AVX-51245 GB/sBW÷45

iGPU / NPU / dGPU 使用 preset compute ceiling，不從 TOPS 推導。C < C_req 時為 Compute-Bound，加 core 直接提升 TPS。

AI Streams 與 Reuse Factor

AI Streams = EBW / BPT_raw
DevicebaseReuse意義

AI Streams 是跨 Intel / AMD / NVIDIA / NPU 的統一指標，不依賴 TOPS。AI Streams 高代表平台能同時服務更多推論請求。

Prefill 模型（CPU）

T_prefill = 2 × P × L / FLOPS

Prefill 是矩陣-向量批次運算，屬 Compute-Bound，與 decode 完全不共用模型。FLOPS 估算：AVX2 ≈ 0.25 TFLOPS、AVX-512 ≈ 1.5 TFLOPS。

條件簡化公式
AVX2P × L ÷ 125
AVX-512P × L ÷ 750

✓ 適用範圍

CPU / iGPU / NPU / dGPU decode/prefill 估算。Batch 1–8。L = 64–8192。x86 與 ARM 架構（reuse 常數需校準）。Intel、AMD、Qualcomm、MTK 等主流平台。

✗ 不適用範圍

Multi-node 分散式推論。量化精度 < INT2。Flash Attention 大幅改變 KV 存取模式的場景。Arrow Lake E-core 密集負載（單一 C 值代入會高估）。誤差超過 ±30%。

使用限制與但書
⚠ 需要注意——可能影響採購判斷
KV Cache 未計入 RAM 估算。 當 L > 2048，模型快取會額外佔用記憶體。以 14B、L=4096 為例，快取約多佔 3–5 GB。建議 L > 2048 時 RAM 估算值加 4 GB。
已知限制（不影響方向，影響數字精度）
FLOPS 固定常數N100 與 i9 共用同一常數，低頻平台 Prefill 偏樂觀 30–50%。
η_numa 未建模雙路 Xeon 跨 socket 效率約打八折，目前未反映。
E-core 混合架構Raptor Lake、Meteor Lake、Arrow Lake 的 P+E 混合架構，本工具以統一 tpc 值估算，P-core 與 E-core 特性不同，可能造成額外誤差。
Attention O(L²)L > 1024 後 Attention 計算量平方成長，Prefill 實際更慢。
Reuse 常數baseReuse 為 heuristic，跨 NPU 世代可能有 ±30% 偏差。
AMD 平台tpc 常數（12/45 GB/s）以 Intel 實測為基準，AMD 方向正確但數字誤差可達 ±20–30%。
本工具定位為 semi-empirical 工程估算器，精度目標 ±15%（CPU），±25%（iGPU/NPU），尚待 benchmark 校準。不適用於多節點分散式推論。
